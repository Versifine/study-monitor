# Exam Monitor 仓库指令

本文件是 Codex 和其他代码代理在本仓库内工作的最高优先级项目规则。任何任务开始前必须完整阅读本文件，并同时阅读任务涉及的 `docs/` 文档。

## 一、产品目的

本项目服务于 2028 年考研准备。主目标是建立一套稳定、低维护、可冻结的个人学习证据记录系统。

正式备考期间，用户不能因为系统不够聪明、出现普通 Bug、模型判断不准确或界面不完善而持续调试和开发。因此，系统价值必须建立在“即使分析全部关闭，原始证据仍能稳定记录”之上。

## 二、不可更改的优先级

1. 不干扰正常学习
2. 不丢失原始证据
3. 自动恢复
4. 支持回滚和关闭单个模块
5. 暴露数据覆盖率和运行健康
6. 分析准确度
7. 功能丰富度和视觉效果

不得为了更聪明的分析、更漂亮的界面或更多功能牺牲稳定性、可恢复性和低维护性。

## 三、Version 1 的定义

Version 1 仅实现 Recorder Core，必须支持：

- 仅追加事件写入
- 外部桌面活动采集器导入
- 外部屏幕录像和摄像头分段文件登记
- 统一时间线
- 采集器心跳
- 数据缺失、延迟、损坏和未完成状态识别
- 本地缓存、幂等写入和断点恢复
- 存储保留策略和磁盘保护
- 自动重启和降级运行
- 备份、恢复和回滚
- 最小只读仪表盘
- 所有 AI 分析关闭时仍可正常运行的 record-only mode

## 四、Version 1 明确排除

除非后续里程碑文档明确要求，不得实现：

- 自动掌握度评分正式上线
- 自由生成考研题并直接用于高权重评价
- 完整手写数学推理分析
- 心理诊断
- 精确饮食热量估算
- 正式版可穿戴摄像头固件
- 人脸识别
- 微服务、Kubernetes、Kafka、Redis 等复杂基础设施
- 多用户系统
- 复杂权限体系
- 原生移动端应用
- 实时逐帧云端 AI 分析
- 以 OCR 文本作为纸面理解主链路

可以保留清晰扩展接口，但不得提前实现排除项。

## 五、架构约束

优先选择最小、可维护的架构：

- 一个 Go 应用承担本地 API、事件接收、健康检查和后台任务
- SQLite WAL 保存结构化事件和元数据
- 本地文件系统保存原始媒体
- 静态 TypeScript 前端构建后嵌入 Go 二进制
- 优先接入 ActivityWatch、FFmpeg 等成熟工具，不重复造采集轮子
- 后续 Python 分析进程必须是可选、隔离、可关闭的
- 正常运行不得依赖 Docker；Docker 只能作为可选开发方式
- Recorder Core 不得依赖云服务才能工作
- Windows 是生产环境一等平台，不得要求 WSL 才能运行

## 六、数据规则

数据分为三层：

1. Evidence：原始视频、屏幕、截图、活动事件、传感器数据
2. Observation：确定性程序或模型对 Evidence 的描述
3. Inference：对学习、健康、心理、饮食等状态的推断

规则：

- Evidence 不可被 Observation 或 Inference 覆盖
- 原始证据默认不可修改，只允许新增状态或保留策略操作
- 每个 Observation 和 Inference 必须保存来源证据引用
- 必须保存生成时间、生产者名称、生产者版本、置信度和状态
- 必须允许未来使用新模型重新分析旧 Evidence
- 内部统一使用 UTC
- 同时保存设备原始时间、服务端接收时间和估计时钟偏移
- 所有写入必须幂等

## 七、学习评估规则

系统不得仅根据观看时长、书页停留或摄像头画面直接声称“已经掌握”。

必须区分：

- 内容覆盖度
- 有效练习量
- 当前表现
- 延迟保持度
- 陌生题迁移度
- 证据独立性
- 结论置信度

视觉模型负责理解外显过程，不得假装读取未写出的脑内推理。掌握度必须主要依赖作答、主动追问、变式题和延迟复测。

## 八、视觉理解规则

纸面理解主链路是：

- 固定俯拍摄像头
- 纸面跟踪与透视校正
- 手部遮挡和翻页检测
- 前后视觉状态比较
- 关键帧和短视频构造
- 视觉语言模型直接理解题目、笔迹变化和解题过程

OCR 只能作为检索、索引或辅助信号，不得成为最终学习判断的唯一依据。

## 九、失败行为

分析失败不得停止采集；单个采集器失败不得停止其他采集器。

模块失败时按以下顺序处理：

1. 有界重试
2. 保留本地缓存
3. 标记覆盖率缺口
4. 进入降级模式
5. 仅在即将丢失数据、磁盘耗尽或严重影响设备性能时通知用户

可选模块出现普通故障时，优先关闭模块，不得要求备考期间立即调试。

## 十、工程规则

- 每次只实现一个里程碑
- 不增加未请求功能
- 不静默扩大范围
- 优先标准库和成熟依赖
- 依赖版本固定
- 必须提供结构化日志
- 必须提供健康检查
- 必须有资源上限和保留策略
- 确定性逻辑必须有单元测试
- 写入、恢复、重复数据和损坏输入必须有集成测试
- 必须提供本地 smoke test
- 行为变化时，同一任务内更新相关文档
- 不得自动进入下一个里程碑

## 十一、目标脚本

仓库最终应具备：

- `scripts/dev.ps1`
- `scripts/test.ps1`
- `scripts/smoke.ps1`
- `scripts/build.ps1`
- `scripts/install.ps1`
- `scripts/uninstall.ps1`
- `scripts/backup.ps1`
- `scripts/restore.ps1`
- `scripts/rollback.ps1`

脚本默认必须非破坏性。

## 十二、里程碑完成条件

只有同时满足以下条件，里程碑才完成：

- 验收标准全部满足
- 测试通过
- smoke test 通过
- 相关文档已更新
- 没有无关改动
- 已审查 diff
- 已列出剩余风险
- 可以安全创建 Git 检查点

完成后必须停止，并汇报改动、执行命令、测试结果、风险和建议的下一里程碑。
