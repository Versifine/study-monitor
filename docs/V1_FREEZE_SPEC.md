# Version 1 冻结规格

## 1. 版本目标

Version 1 是正式备考期间可长期运行的 Recorder Core。

它必须在所有 AI 分析关闭、云端不可用时，仍能稳定保存原始证据、显示数据覆盖率并完成备份恢复。Version 1 的完成线是 M0-M6 全部通过，不是“接口已经为未来分析预留”。

## 2. 必须实现

### 2.1 事件记录

- 仅追加结构化 Evidence 事件
- 批量写入和逐项结果
- 以 `collector_id + idempotency_key` 唯一确定事实
- 相同幂等键和相同内容返回原结果；相同键但不同内容必须报冲突
- 原样保存设备时间表示，并保存归一化 UTC、接收 UTC、估计时钟偏移和偏移误差
- 保存采集器身份、版本和追加式心跳
- 有输入大小、批量大小、并发和重试上限
- 使用稳定排序键和不受新写入扰动的确定性分页

### 2.2 媒体登记

- 只导入外部摄像头和屏幕录制器已经完成的媒体分段，不开发录像器
- 使用已完成媒体、版本化 sidecar 和就绪标记组成导入合同
- 在 Recorder Core 管理的目标卷内先写临时文件，校验后原子改名
- 跨卷来源必须复制、校验和落盘后再确认，不把跨卷移动当作原子操作
- 保存文件大小、时长、设备时间戳、接收时间、校验和和来源采集器
- 未完成文件保持待定；损坏或不合规文件隔离；重复文件幂等确认
- 中断后可从临时文件或来源文件恢复，不删除未知文件和未确认来源文件
- 原始文件删除只能由明确保留策略触发，并保留不可变的删除记录

### 2.3 统一时间线和覆盖率

- 必须支持 ActivityWatch 桌面事件、书桌/屏幕媒体和通用 JSON Evidence 的合并查询
- 手机与健康数据在 Version 1 只能经通用 JSON 合同导入；不开发原生手机端或健康平台连接器
- 时间线返回原始设备时间、归一化 UTC、接收 UTC、校正时间和估计误差，不把校正结果覆盖原时间
- 覆盖率按采集器分别计算，使用左闭右开区间 `[start, end)`
- 可用性状态只取一个：`covered`、`confirmed_idle`、`pending`、`delayed`、`offline`、`unknown`
- 质量另用标记表达：`obscured`、`corrupt`、`clock_uncertain`、`incomplete`
- `confirmed_idle` 只能由仍有有效心跳的采集器明确证明；缺少数据不得推断为空闲
- 迟到数据允许重算覆盖率投影，但必须保留原始事件和计算版本

### 2.4 运维

- 当前用户的 Windows Task Scheduler 任务负责登录后启动和失败后有界重启，不依赖管理员权限、Docker、WSL 或云服务
- 有界重试、指数退避和崩溃循环保护
- 进程内适配器和媒体入口保留本地缓冲；任意客户端仍必须遵守幂等重试合同
- 存储保留策略、磁盘水位和为数据库预留的空间
- 备份、恢复和上一稳定应用版本回滚
- 可选适配器、媒体入口、分析接口和仪表盘可以独立关闭；事件存储和健康检查属于不可关闭的最小核心
- 结构化日志、健康检查、故障记录和资源上限
- 日志按大小/数量轮转，临时文件、WAL 和入口积压都有上限或可证明的收敛策略；日志不是 Evidence，可按运维策略清理

### 2.5 最小界面

只读显示：

- 当日时间线
- 采集器状态
- 数据覆盖率和质量标记
- 存储剩余空间和当前磁盘保护级别
- 媒体导入积压和采集器上报的外部积压
- 分析模块是 `disabled`、`not_installed`、`healthy` 还是 `backlogged`；关闭时不得显示为零积压
- 最近故障、恢复动作和降级状态

仪表盘失败或未打开不得影响写入、恢复和备份。

## 3. 明确不实现

- 完整掌握度正式决策
- 自动调整整体考研计划
- 主动出题、追问和延迟复测业务
- 知识图谱业务
- 纸面视觉理解和实时逐帧视频理解
- 实时心理判断或医学诊断
- 正式可穿戴硬件
- 精确热量统计
- 原生手机应用和原生健康平台连接器
- 人脸识别
- 多用户、复杂登录和远程公网服务
- 微服务、消息中间件、容器编排和云端必需依赖
- 复杂动画与设计系统

Version 1 不为以上功能提前创建业务表、模型队列、Python 运行环境或部署单元。可以保留版本化导入/导出边界。

## 4. 运行模式

### Record-only mode

Version 1 的标准验收模式。只采集、登记、查询、显示健康和执行备份恢复，不运行任何 AI 分析。所有 M0-M6 验收都必须至少在此模式通过。

### Minimum mode

紧急降级模式。只保留 ActivityWatch 桌面事件、书桌媒体登记、核心健康检查和备份；关闭其他适配器、分析接口和非必要界面功能。Minimum mode 必须能够持续运行到初试结束。

“采集、分析和教练能力全部启用”的增强模式属于 Version 1 之后的研究范围，不是 Version 1 模式，也不是完成条件。

## 5. 存储与部署冻结参数

代码必须固定行为合同，具体容量值允许按目标机器配置。以下参数在 M4 必须有安全默认值和关系校验，在 M6 开始前必须写入冻结清单并锁定：

- 每个核心采集器的心跳周期、允许迟到时间、离线判定时间和计划启用时段
- 磁盘预警水位、关键水位和数据库保留空间；必须满足“预警早于关键，关键早于保留空间耗尽”
- 自动保留策略是否启用、最短保留期、可删除数据类别和所需备份状态
- 备份 RPO、恢复 RTO、自动恢复 RTO 和资源上限

安全默认行为是：不自动删除。达到预警水位时关闭可选分析并暂停低优先级导入；达到关键水位时停止接收新媒体，但在数据库保留空间允许时继续写入小型核心事件和故障记录。只有显式启用保留策略后，系统才可删除“由 Recorder Core 管理、已接受、超过最短保留期且满足配置中备份条件”的媒体本体；不得删除未知、写入中、隔离中或未确认的数据。Version 1 不自动删除结构化事件、心跳、故障、媒体元数据或状态历史。

冻结清单在 M6 期间不得修改。测试失败后不得通过放宽阈值继续累计天数。

M6 的核心来源不能配置为空：ActivityWatch 必须覆盖计划的 Windows 登录/测试时段；书桌媒体必须有预先登记的代表性录制窗口。两者在排除事先列明的故障注入窗口后，可用覆盖必须达到计划启用时间的 99% 以上。可用覆盖指不带 `corrupt`、`incomplete` 的 `covered`；书桌媒体还不得带 `obscured`。只有能明确证明 idle 的来源才可把 `confirmed_idle` 计入可用覆盖。ActivityWatch 单次非计划 `offline + unknown` 不得超过 5 分钟；媒体分段时长不得超过 10 分钟，单次非计划无有效媒体不得超过 15 分钟。故障注入窗口不计入 99%，但必须单独满足恢复 RTO，不能从报告中消失。

## 6. 冻结验收

- M0-M5 的验收标准、自动化测试、smoke test 和相关文档全部通过
- 使用同一个候选构建和同一份冻结清单连续运行 14 天；核心行为、依赖、配置或数据 schema 改变后从第 1 天重新开始
- 已确认接收的事件和已接受的媒体由 Recorder Core 造成的数据丢失为 0
- 相同事实重复提交后产生的重复 Evidence 为 0；幂等冲突可见且不覆盖原事实
- 所有启用的核心采集器在结算窗口后，计划启用时间均被归类为一种可用性状态；不存在未被表示的时间洞
- ActivityWatch 与书桌媒体达到第 5 节的最低可用覆盖率和单次缺口上限；不能用持续 `offline` 或空计划通过认证
- 故障注入后在冻结清单的 RTO 内自动恢复或进入文档规定的降级状态，不要求现场改代码
- 数据库完整性检查通过，损坏和未完成媒体不会进入 `accepted` 状态
- 磁盘保护不会占满保留空间，不会自动删除不满足策略的数据
- 关闭所有分析后采集、查询、健康检查和备份恢复正常
- 可选模块可以单独停用；Minimum mode smoke test 通过
- 一条命令完成 smoke test，一条命令验证恢复，一条命令回滚上一稳定应用版本
- 运行状态只按通知门槛打扰用户

外部采集器自身离线或硬件漏录不记为 Recorder Core 数据丢失，但必须在规定时间内准确标记为 `offline`、`delayed` 或 `unknown`，并计入覆盖率报告。

## 7. 冻结后允许的操作

允许：

- 重启服务
- 回滚稳定应用版本
- 关闭故障模块
- 更换损坏硬件
- 执行备份和恢复

禁止：

- 更换技术栈
- 调整模型提示词
- 追求更高识别率
- 增加新设备
- 重构架构
- 开发新功能
- 因普通误判打开 Codex 调试
