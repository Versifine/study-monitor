# 稳定性测试与冻结

## 1. 冻结前测试周期

使用同一个候选构建和同一份冻结清单至少连续运行 14 天，不增加功能、不升级依赖、不放宽阈值。二进制、核心配置、schema、保留行为或采集 SLA 变化后重新从第 1 天累计。

## 2. 冻结清单

开始前记录并校验：

- 应用版本、提交、构建产物和依赖校验和
- 数据库 schema 版本及与上一稳定版本的兼容范围
- 配置文件校验和、启用模块和运行模式
- 每个核心采集器的计划启用时段、心跳周期、允许迟到和离线判定
- 磁盘水位、保留条件、资源上限、备份 RPO、恢复/自动恢复 RTO
- 数据目录、备份目录、上一稳定发布目录和目标 Windows 环境

清单字段缺失或关系校验失败不得开始计时。

## 3. 每日记录指标

- 进程崩溃、自动重启、崩溃循环和降级次数
- 已确认事件数、重复重放数、幂等冲突数和重复 Evidence 数
- 已接受、待定、隔离、损坏和保留删除媒体数；accepted 文件抽样校验结果
- 各核心来源的 `covered`、`confirmed_idle`、`pending`、`delayed`、`offline`、`unknown` 时长
- 覆盖率投影水位、陈旧时长和重建次数
- 数据库完整性、WAL 大小、磁盘增长和备份覆盖范围
- CPU、内存、句柄/协程、队列、WAL、日志和临时目录峰值
- 人工干预、通知、恢复、备份恢复和回滚结果

“外部采集器漏录”和“Recorder Core 丢失已确认数据”必须分开报告。

## 4. 故障注入

在可恢复的测试数据和受控窗口中测试：

- 网络断开
- 核心服务强制终止
- Windows 系统重启
- 文件写入中断和残留临时文件
- 重复事件/媒体提交和幂等键冲突
- 损坏、截断和伪造 sidecar 的媒体
- 模拟磁盘预警、关键和保留空间威胁
- 可选云端不可用（应证明 Recorder Core 无依赖）
- 采集器时钟前跳、后跳和大偏移
- 数据库忙、备份中断、坏备份和不兼容回滚
- 恢复复制中断、删除媒体后状态提交前中断，以及监督进程连续崩溃达到有界停止

故障注入工具不得对真实 Evidence 目录执行破坏性操作；默认使用临时目录或可注入的故障接口。

M4 提供 `.\scripts\fault-injection.ps1` 和扩展后的 `.\scripts\smoke.ps1`。前者覆盖数据库忙/只读、网络隔离、注入磁盘水位和保留/临时文件恢复；后者在唯一临时目录和临时当前用户任务中覆盖进程强杀、崩溃循环、Minimum、完整备份、新目录恢复、操作中断、坏备份和回滚。M6 必须使用冻结候选重跑，不能把 M4 的临时 smoke 代替 14 天认证。

## 5. 通过标准

- Recorder Core 对已确认事件和已接受媒体造成的数据丢失为 0
- 重复 Evidence 为 0；重放和冲突均有可审计结果
- 数据库每日和最终完整性检查通过；`accepted` 元数据与文件一致
- 结算后，启用核心采集器的计划时间 100% 落入一种可用性状态，不存在未表示的时间洞
- 排除事先列明且另按 RTO 验证的故障注入窗口后，ActivityWatch 与书桌媒体的可用覆盖均不低于非空计划启用时间的 99%；ActivityWatch 单次非计划离线/未知不超过 5 分钟，媒体段不超过 10 分钟且单次非计划无有效媒体不超过 15 分钟
- 采集器离线、迟到、损坏和时钟不确定性被正确显示，不被解释为确认空闲
- 每项故障在冻结 RTO 内自动恢复或进入规定降级；计划演练之外无需人工改代码
- 资源和积压不随运行时间无界增长，磁盘保护保住数据库预留空间
- Record-only 始终可用；Minimum mode 和所有可选模块单独停用 smoke 通过
- 从实际备份恢复到新目录后，数据库、配置、媒体清单和抽样媒体校验通过
- 上一稳定应用版本可在不回退数据 schema 的情况下恢复运行
- P0/P1 通知按门槛触发，P2/P3 不频繁打扰用户

任一项失败即为 M6 不通过。不得用改变统计口径、排除失败日期或放宽阈值继续累计。

## 6. 冻结产物

- 固定版本号、构建产物和校验和
- 依赖锁定和 SBOM/依赖清单
- 冻结配置与清单
- 安装、卸载、测试、smoke、备份、恢复和回滚脚本
- 备份恢复与回滚演练证据
- 14 天逐日报告和最终通过/不通过结论
- 运维手册、风险登记和已知问题列表

## 7. 冻结完成定义

系统即使没有达到理想智能程度，只要 M0-M6 全部通过、稳定保存 Evidence 并满足降级要求，就允许冻结。冻结完成后停止开发，不自动进入后续研究里程碑。
