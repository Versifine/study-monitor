# 故障与分心控制策略

所有模块遵循同一顺序：有界重试 → 保留未确认数据 → 标记覆盖缺口/陈旧投影 → 降级或关闭模块 → 仅按 P0/P1 门槛通知。任何失败都不得用无限重试、无限队列或现场改代码作为恢复路径。

## P0：影响正常学习或数据安全

例如 CPU/内存/句柄失控、磁盘保留空间即将耗尽、设备过热、数据库完整性失败或写入确认可能失真。

处理：立即停止故障模块；无法隔离时停止确认新写入，保护现有 Evidence，并通知用户。不得为“继续采集”冒险破坏已有数据或学习环境。

## P1：核心采集或恢复能力长期失效

例如核心采集超过冻结 SLA 离线、受管媒体持续无法登记、备份超过 RPO、核心进程崩溃循环或回滚不可用。

处理：自动重启或回滚；超过 RTO 后进入 Record-only/Minimum mode 或停用该数据源，明确标记覆盖缺口并通知。固定维护窗口内仍未恢复则保持降级。

## P2：分析、报表、投影或部分数据延迟

处理：保留权威事实，标记 `pending`、`delayed` 或投影陈旧，不立即处理。积压有资源上限，超过上限时停止生产更多可选工作，不丢弃 Evidence。

## P3：体验和研究问题

例如：

- 判断不够聪明
- 某题分析错误
- UI 不够漂亮
- 新模型可能更好
- 想接入新设备

处理：考研结束后再开发。

## Version 1 降级矩阵

| 故障 | 必须保留 | 降级行为 | 用户可见状态 |
|---|---|---|---|
| ActivityWatch 不可读 | 已导入事件和断点水位 | 其他来源继续；该来源停止确认 | `offline` 或 `unknown` |
| 通用 JSON 损坏 | 其他合法项和原文件/请求证据 | 逐项拒绝，不阻塞合法项 | `rejected` 与错误码 |
| 媒体仍在写或 sidecar 缺失 | 来源文件 | 保持 `pending`，不导入 | `pending` |
| 媒体校验失败 | 来源文件、隔离副本和原因 | 隔离，不进入 `accepted` | `corrupt` 质量标记/故障 |
| 覆盖率计算失败 | 原始事件、心跳和媒体事实 | 停止投影，时间线原始事实仍可查 | 投影 `stale` |
| 仪表盘关闭或资源失败 | API、采集和运维 | 不注册界面/汇总路由，继续 Recorder Core | `disabled` 或 `unavailable/DASHBOARD_ASSETS_INVALID` |
| 数据库不可写 | 外部缓存、媒体来源和临时文件 | 不确认新写入，停止保留删除 | P0/P1 故障 |
| 磁盘预警 | 已有 Evidence | 停可选分析和低优先级导入 | `warning` |
| 磁盘关键 | 已有 Evidence、数据库保留空间 | 拒绝新媒体，保留小型核心事件 | `critical` |
| 备份失败 | 当前在线数据 | 不替换上一个有效备份 | `backup_overdue`（超过 RPO 为 P1） |
| 应用回滚不兼容 | 当前数据和两个应用版本 | 安全拒绝，进入 Minimum mode | `rollback_blocked` |
| 后续分析/云端失败 | 所有 Evidence | 分析关闭或积压；Recorder Core 不受影响 | `disabled`/`backlogged` |

## M4 可执行故障事实

- `fault_events` 追加保存模块、P0-P3、`active/recovered/degraded/disabled`、稳定错误码、有限详情和 UTC 时间。
- `module_state_events` 追加保存可选模块的 `healthy/degraded/disabled/unavailable` 及原因；自动保留默认以 `RETENTION_DISABLED` 登记。
- 存储从预警/关键/保留水位恢复到正常时追加 `STORAGE_DISK_RECOVERED`，不覆盖旧故障。
- 监督进程跨启动保存崩溃时间窗；超过阈值写本地 `state/supervisor-state.json` 的 `SUPERVISOR_CRASH_LOOP`，Task Scheduler 与监督层都不会无限重启。
- 媒体在磁盘预警及以上返回 `MEDIA_STORAGE_PROTECTED`；每个候选、每个 1 MiB 复制块和原子改名前都直接刷新系统可用空间，保留来源/暂存未确认数据。数据库保留空间受威胁时，HTTP 与 ActivityWatch/媒体元数据在实际提交前同样实时探测并停止确认，API 返回 HTTP 507；30 秒后台扫描只负责周期状态和故障事实，不是提交门控的数据源。
- 覆盖率失败只在状态转移时追加 degraded 故障和模块状态，后续成功追加 `COVERAGE_RECOVERED`/healthy；权威事实和其他读取不受投影失败影响。
- 保留删除先追加 `planned`；文件删除后，`deleted`、媒体 `retention_deleted` 和投影在同一事务提交。事务失败仍保持可重试的 `planned`，普通缺失不能伪装成保留删除。

## M5 界面故障隔离

- 静态资源在启动时一次性校验并读取；任一资源缺失或为空会关闭整个仪表盘处理器，不提供不完整页面。
- 汇总、时间线或覆盖率读取失败只在页面内显示稳定错误/无数据，不改变写入 readiness，也不重试成无限轮询。
- 浏览器请求有 8 秒等待上限，页面只按用户刷新串行读取；关闭页面不会停止或取消后台采集。
- Minimum mode 与 `EXAM_MONITOR_DASHBOARD_ENABLED=false` 是明确关闭，不是假报健康；分析始终为 `not_installed`，不产生隐藏队列。
- 静态资源和汇总仅接受 GET/HEAD，页面无表单；浏览器不能借 M5 修改 Evidence、保留、计划、模式或模型状态。

## 必须接受的简单替代

- 视觉分析关闭后只保存视频
- 自动出题失败后只用验证题库（后续里程碑）
- 掌握度模型失效后只显示事实指标（后续里程碑）
- 胸前设备失效后保留桌面和书桌采集（后续硬件）

复杂能力没有简单替代方案时不得进入正式备考运行路径。
