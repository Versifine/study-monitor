# 风险登记

## 五个最高风险

| 排名 | 风险 | 触发/证据 | 影响 | Version 1 缓解 | 退出条件 |
|---:|---|---|---|---|---|
| 1 | 媒体增长、保留或备份语义错误造成磁盘耗尽/证据丢失 | 增长无界、关键水位未触发、删除对象不满足策略、备份只含索引却被当成完整备份 | 影响学习设备或不可恢复地丢 Evidence | 受管媒体所有权；默认不自动删除；预警/关键/保留空间三级保护；备份覆盖范围显式化 | 低磁盘、保留、完整恢复故障注入全部通过 |
| 2 | Windows 自动恢复和应用回滚无法形成真实闭环 | 进程崩溃后需人工、任务重启循环、schema 与上一版不兼容 | 正式备考期间被迫现场开发 | 当前用户 Task Scheduler；有界退避；schema 兼容窗口；回滚不做 down migration | M4/M6 在目标 Windows 上完成进程终止、系统重启和回滚演练 |
| 3 | 外部采集器在确认前没有可靠缓存 | ActivityWatch/录像器在 Core 离线时覆盖或删除尚未确认的数据 | Evidence 在到达 Recorder Core 前丢失，Core 无法恢复 | 冻结来源缓存、幂等键、确认和清理合同；来源只在确认后释放数据 | M3/M4 断网、停进程和积压恢复测试证明未确认数据可重放 |
| 4 | 多设备时间和覆盖率把离线误判为学习/空闲 | 心跳缺失仍显示 idle、时钟偏移后事件顺序被伪造、迟到数据不重算 | 学习事实统计失真且难以察觉 | 双时间戳、偏移误差、按来源覆盖状态、质量标记、无心跳禁止 confirmed_idle | M3 固定数据与 M6 实测覆盖报告通过边界/迟到/时钟故障测试 |
| 5 | 后续智能范围渗入 Recorder Core | 提前出现 Observation/Inference/知识图谱表、分析队列、Python/云依赖或原生移动端 | 增加故障面、维护成本并破坏 record-only | V1 架构只含 Evidence；扩展实体不建表；每里程碑检查排除项和目录 | M0-M6 diff 无后续模块，Record-only 离线完成全部验收 |

## 其他已知风险

| 风险 | 影响 | 缓解 |
|---|---|---|
| 进度管理（当前较低） | 若未来自行设定硬截止日，可能诱发跳过验收 | 当前时间充裕；以 M0-M6 质量门槛为准，完整保留 14 天认证，不把 2026 年 9 月当作强制上线日 |
| 视觉模型误判 | 错误评估 | 后续保留原始证据、置信度和拒绝判断；不进入 V1 |
| 题目质量不足 | 错误测评 | 后续题目分级与验证；不进入 V1 |
| 采集器离线被当成无活动 | 错误统计 | 追加心跳、来源 SLA 和覆盖率状态 |
| 云端模型不可用 | 后续分析中断 | Recorder Core 不依赖云端；后续异步、可关闭 |
| 硬件佩戴不现实 | 设备弃用 | 后续先做哨兵原型；不作为 V1 依赖 |
| 系统改变用户行为 | 指标焦虑 | 只读事实和周级趋势，避免实时总分 |
| 敏感本地 API 暴露或被网页滥用 | 私人 Evidence 泄露/伪造 | 默认只监听 loopback；严格 JSON 与 Origin 策略；非本机监听不作为 V1 验收路径 |
| Codex 使调试过于容易 | 备考分心 | 源码隔离、固定维护窗口、只运行冻结构建产物 |

## M4 关闭证据与剩余风险（2026-07-22）

- 已关闭到 M4 门槛：注入 `normal/warning/critical/reserve`、默认保留关闭、完整备份覆盖校验、删除后状态提交前恢复、数据库忙/只读、网络隔离、进程强杀与崩溃循环、备份/恢复中断、坏备份、兼容/不兼容回滚、当前用户 Task Scheduler 注册/卸载均由测试或真实 smoke 覆盖。
- 仍由 M6 承担：真实 Windows 重启后的登录触发、冻结候选连续 14 天资源曲线、真实备份介质 RPO/RTO、整库与媒体抽样回放、目标机器电池/睡眠/杀毒软件影响。
- 当前保留删除仅在显式启用且磁盘不为 `normal` 时执行；M6 冻结前必须演练并确认阈值，正式默认仍保持关闭。
- 本地备份 manifest 提供完整性和操作防错，不提供对拥有本机写权限攻击者的密码学真实性；跨信任域签名和云备份不在 Version 1。
